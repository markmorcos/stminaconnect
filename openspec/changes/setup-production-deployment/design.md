## Context

Solo developer, low budget, free-tier-first, EU-region requirement. Production deployment must be fast to repeat (we'll do it many times after v1 with new features) and cheap to run. The risk shape: small user base, but irreversible actions like accidental migrations or deletions deserve runbook discipline.

## Goals

- Reproducible production setup from a checklist.
- Same migrations, same Edge Functions across local and production.
- Daily backup with off-Supabase weekly archive.
- TestFlight (iOS) and direct APK (Android) distribution.
- Documented incident response.

## Non-Goals

- Public app-store listing. We use TestFlight + internal APK by design.
- Multi-region deployment.
- Blue/green deploys, canary traffic shifting. Single environment.
- CI/CD automation. Manual deploys via Makefile remain the ops model.
- Observability beyond Supabase's built-in logs and our small `logs` table.

## Decisions

1. **Hosted Supabase project**, EU/Frankfurt region. Free tier confirmed sufficient for our row counts and bandwidth profile.
2. **Migrations applied via Supabase CLI** against production:
   - `supabase link --project-ref <prod-ref>`
   - `supabase db push` (preview first, apply second).
3. **Edge Functions deployed individually** via `supabase functions deploy <name> --project-ref <prod-ref>`. Makefile loops over `supabase/functions/*` directories.
4. **Edge Function secrets** set via `supabase secrets set` referencing a runbook-managed `.env.secrets` file (gitignored) for one-shot operations.
5. **Auth redirect URLs**: production scheme `stminaconnect://auth/callback` added to Supabase Auth's allow-list. Dev URLs removed from prod project to avoid leakage.
6. **Backup strategy** (resolves Open Question H2):
   - Supabase daily automatic backups: 7-day retention.
   - Weekly `pg_dump` via `weekly-backup` Edge Function uploaded to Backblaze B2 (free tier 10GB).
   - Off-Supabase retention 90 days (manually rotated).
   - Restore drill: documented in `docs/runbook.md`, performed once per quarter.
7. **EAS production builds**:
   - iOS: signed via paid Apple Developer team (admin enrolls). Provisioning automated by EAS.
   - Android: keystore generated by EAS; key references stored in EAS secrets.
   - Production env vars wired in `eas.json` `production` profile (Supabase prod URL, anon key, dispatcher=`real`).
8. **TestFlight distribution**:
   - `eas submit --profile production --platform ios` uploads to App Store Connect.
   - Internal testers added by email; no public TestFlight beta in v1.
9. **Android internal distribution**:
   - `eas build --profile production --platform android` produces an .aab.
   - Side-load via APK derived from .aab (or use `--profile preview` for direct APK).
   - Distribute via EAS-hosted install URL (QR code) shared in church WhatsApp / email.
10. **Versioning**: `1.0.0` for the first production build. Subsequent releases bump per semver. The `runtimeVersion` policy is `sdkVersion`, so OTA updates within an SDK version are possible (post-v1 enhancement).
11. **Observability**: built-in Supabase Logs Explorer for backend; our `logs` table for app-side errors; manual review weekly. No third-party APM in v1.
12. **Incident response**: small. Document `docs/incident-response.md` with three categories — auth outage, sync outage, push outage — each with first-five-minute checks (Supabase status page, recent migrations, recent Edge Function deploys, recent app submission).

13. **Bundle identifier (final)**: locked to `com.stminaconnect` for both iOS `bundleIdentifier` and Android `package`. Set in `init-project-scaffolding`, confirmed in `prepare-store-listings`, and used here for App Store Connect / Play Console app records. Reverse-DNS uses the developer's domain `stminaconnect.com`.

14. **Apple Developer enrolment**: paid Apple Developer Program membership is required before this phase can run. Solo developer enrols personally; the church does not need its own team membership in v1. Tracked as a v1 release gate; documented in `docs/runbook.md`.

15. **Legal review of Privacy Policy and Terms**: drafts produced in `add-gdpr-compliance` are accepted for internal pilot via TestFlight + APK distribution. **Public store submission requires qualified legal review** of all three language versions. The runbook records this as a release gate; submission tasks are blocked until review is signed off.

16. **Right-to-Erasure runbook entry**: GDPR Article 17 hard-erasure (the `add-gdpr-compliance` admin path) is the canonical erasure mechanism. The runbook documents:
    - The in-app admin path (Compliance screen → typed-confirmation + reason).
    - The backup-retention implication: erased data persists in Supabase native backups (7d) and Backblaze B2 weekly archives (90d). For requests requiring removal from backups before retention expires, document the manual procedure (Supabase support ticket + B2 object deletion).
    - The audit log is immutable; erasure events remain logged.

## Risks / Trade-offs

- **Risk**: free-tier Supabase has bandwidth caps. Mitigation: monitor monthly usage; <200 members is way below the cap.
- **Risk**: Backblaze B2 is a third-party — single point of failure for off-site backup. Mitigation: documented; can swap to AWS S3 if needed.
- **Risk**: TestFlight invite cap is 100 internal testers — well above need.
- **Trade-off**: no CI. Solo dev workflow. Documented as deferred.
- **Trade-off**: no automated rollback. Down migrations must be written for every up migration; phase 1+ already establishes that pattern.
