# API Design

## Overview

The API surface uses three layers:

1. **Supabase PostgREST** — Auto-generated CRUD endpoints for all tables, secured by RLS
2. **Postgres RPC Functions** — Custom queries called via `supabase.rpc()` for complex operations
3. **Edge Functions** (Deno) — Server-side logic for external integrations (Google Calendar, push notifications)

All requests require a valid JWT from Supabase Auth (passed as `Authorization: Bearer <token>`).

---

## Authentication

### Phone OTP Login

**Step 1: Request OTP**

```
POST /auth/v1/otp
Content-Type: application/json

{ "phone": "+491234567890" }

→ 200 OK (OTP sent via SMS)
```

**Step 2: Verify OTP**

```
POST /auth/v1/verify
Content-Type: application/json

{
  "phone": "+491234567890",
  "token": "123456",
  "type": "sms"
}

→ 200 OK
{
  "access_token": "eyJ...",
  "refresh_token": "abc...",
  "user": { "id": "uuid", "phone": "+49..." }
}
```

**Step 3: Refresh Token**

```
POST /auth/v1/token?grant_type=refresh_token
Content-Type: application/json

{ "refresh_token": "abc..." }

→ 200 OK { "access_token": "...", "refresh_token": "..." }
```

### Authorization Model

| Role    | Access                                                                                    |
| ------- | ----------------------------------------------------------------------------------------- |
| servant | CRUD own assigned persons, own follow-ups, all attendance (mark), read all persons/events |
| admin   | Full CRUD on everything. Manage servants. Configure alerts. Delete persons (GDPR).        |

RLS enforces this at the database level. The app never bypasses RLS — it uses the anon key + user JWT.

## PostgREST Endpoints (Auto-generated by Supabase)

All tables are accessible via standard REST. The Supabase JS client abstracts these.

### Servants

```
GET    /rest/v1/servants                  — List all servants
GET    /rest/v1/servants?id=eq.{id}       — Get servant by ID
PATCH  /rest/v1/servants?id=eq.{id}       — Update own profile (push_token, language)
```

**Client usage**:

```typescript
// Get current user's profile
const { data } = await supabase
  .from("servants")
  .select("*")
  .eq("id", session.user.id)
  .single();

// Update push token
await supabase.from("servants").update({ push_token: token }).eq("id", session.user.id);
```

### Persons

```
GET    /rest/v1/persons                           — List all persons (RLS filters per role)
GET    /rest/v1/persons?assigned_servant_id=eq.{id} — My assigned members
GET    /rest/v1/persons?id=eq.{id}                — Get person by ID
POST   /rest/v1/persons                           — Create person (Quick Add or Full)
PATCH  /rest/v1/persons?id=eq.{id}                — Update person
```

## RPC Functions (Postgres)

These run inside Postgres for performance and atomicity. Called via `supabase.rpc()`.

### `get_my_group(servant_id uuid)`

Returns the servant's assigned persons with attendance summary.

```sql
CREATE OR REPLACE FUNCTION get_my_group(p_servant_id uuid)
RETURNS TABLE (
  id uuid,
  first_name text,
  last_name text,
  phone text,
  status person_status,
  priority priority_level,
  last_attended_date date,
  consecutive_misses integer
) AS $$
  SELECT
    p.id, p.first_name, p.last_name, p.phone, p.status, p.priority,
    (SELECT MAX(a.event_date) FROM attendance a WHERE a.person_id = p.id AND a.present = true),
    -- consecutive misses calculated from counted events
    COALESCE((
      SELECT COUNT(*)::integer
      FROM (
        SELECT a.event_date
        FROM attendance a
        JOIN alert_config ac ON true
        WHERE a.person_id = p.id
          AND a.present = false
          AND a.event_title = ANY(ac.counted_event_patterns)
          AND NOT EXISTS (
            SELECT 1 FROM attendance a2
            WHERE a2.person_id = p.id
              AND a2.present = true
              AND a2.event_date > a.event_date
          )
        ORDER BY a.event_date DESC
      ) sub
    ), 0)
  FROM persons p
  WHERE p.assigned_servant_id = p_servant_id
  ORDER BY p.last_name, p.first_name;
$$ LANGUAGE sql STABLE SECURITY DEFINER;
```

### `get_dashboard_stats()`

Admin-only. Returns overview cards data.

```sql
CREATE OR REPLACE FUNCTION get_dashboard_stats()
RETURNS json AS $$
  SELECT json_build_object(
    'total_members', (SELECT COUNT(*) FROM persons),
    'newcomers_this_month', (
      SELECT COUNT(*) FROM persons
      WHERE registered_at >= date_trunc('month', now())
    ),
    'avg_attendance_rate', (
      SELECT ROUND(
        COUNT(*) FILTER (WHERE present = true)::numeric /
        NULLIF(COUNT(*)::numeric, 0) * 100, 1
      )
      FROM attendance a
      JOIN alert_config ac ON a.event_title = ANY(ac.counted_event_patterns)
      WHERE a.event_date >= now() - interval '8 weeks'
    ),
    'at_risk_count', (
      SELECT COUNT(*) FROM follow_ups
      WHERE status = 'pending' AND reason = 'absence_alert'
    ),
    'active_members', (SELECT COUNT(*) FROM persons WHERE status = 'active'),
    'inactive_members', (SELECT COUNT(*) FROM persons WHERE status = 'inactive')
  );
$$ LANGUAGE sql STABLE SECURITY DEFINER;
```

### `get_attendance_for_event(p_event_id text, p_event_date date)`

Returns all attendance records for an event, used on the check-in screen.

```sql
CREATE OR REPLACE FUNCTION get_attendance_for_event(
  p_event_id text,
  p_event_date date
)
RETURNS TABLE (
  person_id uuid,
  first_name text,
  last_name text,
  present boolean,
  marked_by uuid
) AS $$
  SELECT p.id, p.first_name, p.last_name,
    COALESCE(a.present, false),
    a.marked_by
  FROM persons p
  LEFT JOIN attendance a
    ON a.person_id = p.id
    AND a.google_event_id = p_event_id
    AND a.event_date = p_event_date
  ORDER BY p.last_name, p.first_name;
$$ LANGUAGE sql STABLE SECURITY DEFINER;
```

### `bulk_upsert_attendance(records jsonb)`

Sync endpoint for offline attendance. Uses upsert to handle conflicts.

```sql
CREATE OR REPLACE FUNCTION bulk_upsert_attendance(records jsonb)
RETURNS integer AS $$
DECLARE
  affected integer := 0;
  rec jsonb;
BEGIN
  FOR rec IN SELECT * FROM jsonb_array_elements(records)
  LOOP
    INSERT INTO attendance (
      person_id, google_event_id, event_title, event_date,
      present, marked_by, marked_at, synced_at
    ) VALUES (
      (rec->>'person_id')::uuid,
      rec->>'google_event_id',
      rec->>'event_title',
      (rec->>'event_date')::date,
      (rec->>'present')::boolean,
      (rec->>'marked_by')::uuid,
      (rec->>'marked_at')::timestamptz,
      now()
    )
    ON CONFLICT (person_id, google_event_id, event_date)
    DO UPDATE SET
      present = EXCLUDED.present,
      marked_by = EXCLUDED.marked_by,
      marked_at = EXCLUDED.marked_at,
      synced_at = now()
    WHERE attendance.marked_at < EXCLUDED.marked_at;  -- last-write-wins

    affected := affected + 1;
  END LOOP;

  RETURN affected;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

### `get_absence_streaks()`

Returns persons with consecutive misses at or above threshold. Used by `check-absences` Edge Function.

```sql
CREATE OR REPLACE FUNCTION get_absence_streaks()
RETURNS TABLE (
  person_id uuid,
  person_name text,
  assigned_servant_id uuid,
  consecutive_misses integer,
  threshold integer,
  last_attended date
) AS $$
  WITH config AS (
    SELECT * FROM alert_config LIMIT 1
  ),
  counted_attendance AS (
    SELECT a.person_id, a.event_date, a.present
    FROM attendance a, config c
    WHERE a.event_title = ANY(c.counted_event_patterns)
  ),
  streaks AS (
    SELECT
      p.id AS person_id,
      p.first_name || ' ' || p.last_name AS person_name,
      p.assigned_servant_id,
      COUNT(*) FILTER (
        WHERE ca.present = false
        AND NOT EXISTS (
          SELECT 1 FROM counted_attendance ca2
          WHERE ca2.person_id = p.id
            AND ca2.present = true
            AND ca2.event_date > ca.event_date
        )
      )::integer AS consecutive_misses,
      COALESCE(
        (config.priority_thresholds->>p.priority::text)::integer,
        config.default_threshold
      ) AS threshold,
      MAX(ca.event_date) FILTER (WHERE ca.present = true) AS last_attended
    FROM persons p
    CROSS JOIN config
    LEFT JOIN counted_attendance ca ON ca.person_id = p.id
    WHERE p.paused_until IS NULL OR p.paused_until < CURRENT_DATE
    GROUP BY p.id, p.first_name, p.last_name, p.assigned_servant_id, p.priority,
             config.priority_thresholds, config.default_threshold
  )
  SELECT * FROM streaks
  WHERE consecutive_misses >= threshold;
$$ LANGUAGE sql STABLE SECURITY DEFINER;
```

### `get_weekly_attendance_trend(weeks integer DEFAULT 12)`

Returns weekly attendance counts for the admin chart.

```sql
CREATE OR REPLACE FUNCTION get_weekly_attendance_trend(p_weeks integer DEFAULT 12)
RETURNS TABLE (
  week_start date,
  total_present integer,
  total_expected integer
) AS $$
  SELECT
    date_trunc('week', a.event_date)::date AS week_start,
    COUNT(*) FILTER (WHERE a.present = true)::integer,
    COUNT(*)::integer
  FROM attendance a
  JOIN alert_config ac ON a.event_title = ANY(ac.counted_event_patterns)
  WHERE a.event_date >= CURRENT_DATE - (p_weeks || ' weeks')::interval
  GROUP BY 1
  ORDER BY 1;
$$ LANGUAGE sql STABLE SECURITY DEFINER;
```

## Edge Functions (Deno)

Deployed to Supabase Edge Functions. Called via `https://<project>.supabase.co/functions/v1/<name>`.

### `fetch-events`

**Trigger**: pg_cron every 30 minutes, or manually via admin action.

**Purpose**: Pull events from Google Calendar API and cache them locally.

```
POST /functions/v1/fetch-events
Authorization: Bearer <service_role_key>

→ 200 OK { "fetched": 5, "upserted": 3 }
```

**Logic**:

1. Authenticate with Google Calendar API using service account key (from env)
2. Fetch events from configured calendar ID for next 7 days + past 3 days
3. Upsert into `cached_events` table
4. Return count of fetched/upserted events

**Environment Variables**:

- `GOOGLE_SERVICE_ACCOUNT_KEY` — JSON key for Google service account
- `GOOGLE_CALENDAR_ID` — Calendar ID to fetch from

### `check-absences`

**Trigger**: Called after attendance sync completes (from client via webhook or pg_cron after each service day).

```
POST /functions/v1/check-absences
Authorization: Bearer <service_role_key>

→ 200 OK
{
  "checked": 45,
  "alerts_created": 2,
  "notifications_sent": 2,
  "welcome_backs": 1
}
```

**Logic**:

1. Call `get_absence_streaks()` RPC to find persons above threshold
2. For each flagged person:
   a. Check if a `pending` follow-up already exists → skip if yes
   b. Create new `follow_up` with reason `absence_alert`
   c. Call `send-notification` for assigned servant
   d. If `notify_admin` is true in `alert_config`, notify admins too
3. **Return detection**: Find persons who were flagged (have pending absence follow-ups) but attended today
   a. Send "Welcome back" notification to assigned servant
   b. Auto-complete the pending follow-up with notes "Returned to attendance"

### `send-notification`

**Trigger**: Called by `check-absences` or directly for manual notifications.

```
POST /functions/v1/send-notification
Authorization: Bearer <service_role_key>
Content-Type: application/json

{
  "servant_id": "uuid",
  "title": "Absence Alert",
  "body": "Mina Ibrahim has missed 3 consecutive Sunday Liturgies",
  "data": {
    "type": "absence_alert",
    "person_id": "uuid",
    "follow_up_id": "uuid"
  }
}

→ 200 OK { "sent": true, "ticket_id": "..." }
```

**Logic**:

1. Look up `push_token` from `servants` table
2. If no token → log warning, return `{ "sent": false, "reason": "no_push_token" }`
3. POST to Expo Push API: `https://exp.host/--/api/v2/push/send`
4. Handle receipts for failed deliveries (retry once)

## Sync API

The client doesn't call a dedicated sync endpoint. Instead:

### Push (Client → Server)

1. Client reads `sync_queue` from local SQLite
2. Groups by table: persons, attendance, follow_ups
3. Calls the appropriate PostgREST endpoint or RPC:
   - Persons: `POST /rest/v1/persons` or `PATCH` (upsert via ON CONFLICT)
   - Attendance: `supabase.rpc('bulk_upsert_attendance', { records })`
   - Follow-ups: `PATCH /rest/v1/follow_ups`
4. On success, marks `sync_queue` items as synced

### Pull (Server → Client)

1. Client calls:
   - `GET /rest/v1/persons?updated_at=gt.{last_synced}` (for admin: all; for servant: all, but could optimize)
   - `GET /rest/v1/attendance?synced_at=gt.{last_synced}`
   - `GET /rest/v1/follow_ups?updated_at=gt.{last_synced}` (servant: own only)
   - `GET /rest/v1/cached_events?date=gte.{today-3}`
2. Upserts results into local SQLite
3. Updates `sync_meta.last_synced_at` per table

## Error Handling

| Scenario        | Response                         | Client Behavior                                 |
| --------------- | -------------------------------- | ----------------------------------------------- |
| Invalid JWT     | 401 Unauthorized                 | Redirect to login, clear session                |
| RLS violation   | 403 Forbidden                    | Show error toast, log for debugging             |
| Duplicate phone | 409 Conflict (unique constraint) | Show "Phone number already registered"          |
| Network error   | Timeout / no response            | Keep in sync_queue, retry next cycle            |
| Rate limit      | 429 Too Many Requests            | Exponential backoff (handled by TanStack Query) |
| Server error    | 500                              | Show generic error, retry                       |

## Rate Limiting & Security

| Measure          | Implementation                                                        |
| ---------------- | --------------------------------------------------------------------- |
| Rate limiting    | Supabase built-in: 100 req/sec per project (free tier)                |
| Input validation | Zod schemas on client. Postgres constraints on server.                |
| SQL injection    | Parameterized queries via Supabase client (never raw SQL from client) |
| CORS             | Supabase configured to allow only the app's origin                    |
| JWT expiry       | Access tokens expire in 1 hour. Refresh token rotation enabled.       |
| Service role key | Used only in Edge Functions (server-side). Never exposed to client.   |
| Phone validation | E.164 format enforced by Zod schema before submission                 |
